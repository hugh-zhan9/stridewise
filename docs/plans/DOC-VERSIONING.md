# 设计文档版本管理规范（DOC-VERSIONING）

## 1. 适用范围
适用于 `docs/plans/` 下所有设计类文档，包括：
- 详细设计
- 开发计划
- OpenAPI 规范
- JSON Schema 规范
- 其他技术方案文档

## 2. 版本号格式
统一使用三段式：`主版本.次版本.修订号`（如 `v2.1.3`）

- 主版本（Major）：`X`
- 次版本（Minor）：`Y`
- 修订号（Patch）：`Z`

## 3. 升版规则
### 3.1 主版本 +1（Major）
以下任一情况触发：
- 架构范式变化（如规则引擎改为 AI+规则混合）
- 核心数据模型不兼容变更
- 接口契约发生破坏性变更（不兼容旧客户端）

处理：
- `vX.Y.Z -> v(X+1).0.0`

### 3.2 次版本 +1（Minor）
以下任一情况触发：
- 新增模块/新章节/新能力（向后兼容）
- 新增 API、Schema 字段（兼容式）
- 新增关键流程、异常策略、可观测指标

处理：
- `vX.Y.Z -> vX.(Y+1).0`

### 3.3 修订号 +1（Patch）
以下任一情况触发：
- 文案澄清、错字修复
- 不改变语义的表述优化
- 轻量补充说明（不影响实现边界）

处理：
- `vX.Y.Z -> vX.Y.(Z+1)`

## 4. 文档头部模板（必须）
每份文档顶部必须包含：

```markdown
## 文档版本
- 当前版本：vX.Y.Z
- 发布日期：YYYY-MM-DD
- 文档状态：草稿/可评审/已确认

## 变更记录
| 版本号 | 日期 | 变更说明 |
| --- | --- | --- |
| vX.Y.Z | YYYY-MM-DD | 本次变更摘要 |
```

## 5. 修改标注规则（必须）
### 5.1 删除线规则
- 对“被替换的原句”使用删除线：`~~原内容~~`
- 删除线后紧跟新内容，便于评审比对

示例：
- ~~文档关系：本文件是接口补充~~
- 文档关系：本文件是详细设计附录，包含 OpenAPI 与 Schema。

### 5.2 大段改写规则
- 若章节整体重写，不要求对每行加删除线。
- 需在“变更记录”中明确写清“重写章节范围与原因”。

## 6. 关联文档版本联动
- 主设计文档升级 `Major/Minor` 时，附录文档必须同步检查并更新版本。
- 附录发生契约变更（API/Schema）时，主设计需在“关联文档”处同步记录。

建议联动策略：
- 主设计：`vX.Y.Z`
- 附录文档：独立版本号，但在变更说明中注明“对应主设计版本”。

## 7. 提交流程（建议）
1. 修改文档内容
2. 更新版本号与发布日期
3. 按规则标注删除线
4. 更新变更记录
5. 自检关联文档是否需要联动升版
6. 提交变更

## 8. 自检清单
- 是否包含“文档版本”区块
- 是否包含“变更记录”区块
- 是否按升版规则更新版本号
- 是否对替换内容使用删除线
- 是否检查关联文档联动
- 变更说明是否足够让他人理解影响范围
