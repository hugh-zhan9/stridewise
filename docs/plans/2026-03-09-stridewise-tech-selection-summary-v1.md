## 1. 产品形态
- 首期形态：Web 应用
- 后续方向：可扩展为 App
- 架构方式：前后端分离，Web/App 共用后端 API

## 2. 后端技术栈
- Web 框架：Kratos
- ORM：Ent
- 数据库：PostgreSQL（不使用 SQLite）
- 异步队列：Asynq
- 队列中间件：Redis

## 3. 任务与调度策略
- 必须使用异步任务处理采集/标准化等耗时流程
- 异步任务字段强制要求：
  - `job_id`
  - `status`
  - `retry_count`
  - `error_message`
- 定时任务策略：
  - 当前为单节点部署，先采用轻量定时方式（进程内 cron 或 Asynq scheduler）
  - 暂不引入复杂分布式调度框架

## 4. 安全与凭证策略
- 当前阶段不引入完整用户鉴权体系（仅适用于单用户/早期内测阶段）
- ~~当前阶段无外部 API，全部接口按内部接口治理~~
- 当前阶段对外无公开 API，全部业务接口按内部接口治理；允许向 OpenAI 发起外部调用
- 第三方采集凭证（token）必须加密存储
- 严禁明文落库凭证信息

## 5. 数据层原则
- 以标准化活动模型作为核心数据源
- 保留原始采集数据与任务状态，满足审计、重试、回溯
- 通过配置文件指定运行配置（当前阶段）
- 第三方数据源接入顺序：P0 首发 Keep，后续扩展 Strava/Garmin/Nike/GPX/TCX/FIT

## 6. 阶段性边界与后续触发条件
- 现阶段“无鉴权”仅适用于单用户或受控环境
- 当进入多用户或 App 外部开放阶段，必须补齐：
  - 用户鉴权与授权体系
  - 数据隔离机制
  - Token 生命周期管理与撤销能力

## 7. 部署方案（已确认）
- 部署方式：Docker Compose 一键部署
- 部署形态：单节点（当前阶段）
- 服务编排建议：
  - `api`（Kratos 应用）
  - `worker`（Asynq 消费者）
  - `scheduler`（定时任务进程，可与 worker 合并）
  - `postgres`（PostgreSQL）
  - `redis`（Asynq 队列）
- 架构与开发约束：
  - 本地开发、测试、生产尽量保持同构（都通过 Compose 运行）
  - 配置项必须支持容器化环境变量与配置文件双通道
  - 服务健康检查、重启策略、日志落盘路径需在 Compose 中显式定义
