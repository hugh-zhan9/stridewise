## 1. 开发目标
- 4-6 周完成 P0 功能，达到内测上线标准。
- 确保“安全优先、建议可解释、异常可兜底、指标可观测”。

## 2. 迭代排期（6周）

### 第1周：基础设施与数据骨架
任务：
- 建库建表（user_profile/training_log/weather_snapshot/recommendation/feedback/baseline_history）
- ~~搭建 API 基础框架与鉴权~~

- 可运行后端骨架

### 第4周：训练后总结与反馈闭环

## 5. 上线计划

### 5.0 部署基线（已确认）
- 部署方式：Docker Compose 一键部署
- 部署形态：单节点
- 运行组件：`api`、`worker`、`postgres`、`redis`（`scheduler` 可并入 `worker`）
- 鉴权现状：当前阶段不启用完整用户鉴权；内部接口使用 `X-Internal-Token`

### 5.1 内测上线门槛（Go/No-Go）
1. P0 功能完成率 100%
2. 安全规则拦截正确率 >= 99%
3. 建议链路可用性 >= 99.5%
4. 关键异常回归通过率 100%

### 5.2 灰度策略
- 灰度用户占比：10%-20%
- 核心监控指标：
  - 建议采纳率
  - 训练完成率
  - 不适反馈率
  - 建议生成失败率

### 5.3 回滚策略
- 规则异常：回滚到上个 `engine_version`
- 天气异常：强制切换保守建议模板

## 6. 风险与应对
1. 天气数据波动
- 应对：多源备份 + 快照回退

2. 第三方导入不稳定
- 应对：重试队列 + 手动补录

3. 建议可信度不足
- 应对：强制解释字段 + 反馈迭代

4. 合规风险
- 应对：上线前法务审查 + 明示免责声明
