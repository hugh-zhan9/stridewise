# 夜间基线重算设计 v1.0.0

> **版本记录**
> - v1.0.0 (2026-03-11)：首版，服务器时区 02:00 触发，仅覆盖近 28 天有训练/活动的用户。
> - v1.1.0 (2026-03-11)：增加 02:00 后启动补跑与执行记录表。

## 1. 目标与范围

**目标**
- 每日凌晨 02:00（服务器时区）触发基线重算。
- 02:00 后启动时补跑一次（仅当当天尚未执行）。
- 仅对最近 28 天内有训练记录或导入活动的用户执行。
- 复用已有 `baseline:recalc` 异步任务逻辑。
- 记录每日执行状态，避免重复执行。

**非目标**
- 不引入独立 scheduler 服务。
- 不进行复杂分区/批处理优化。

## 2. 触发策略

- ~~触发时间：服务器时区每日 02:00。~~
- 触发时间：服务器时区每日 02:00；worker 启动后如已过 02:00 且当天未执行，则立即补跑一次。
- 触发范围：
  - 最近 28 天内有 `training_logs` 或 `activities` 记录的用户。
  - 用户集合去重后逐一入队。

## 3. 执行记录

- 新增 `nightly_baseline_runs` 记录每日执行状态（日期、开始/完成时间、状态、错误信息）。
- 若当天已执行成功，后续触发直接跳过，避免重复入队。

## 4. 入队策略

- 任务类型：`baseline:recalc`（沿用现有实现）。
- `trigger_type = "nightly"`
- `trigger_ref = "nightly-YYYYMMDD"`

## 5. 实现位置

- 新增 `RunNightlyBaselineRecalc(ctx, store, enqueuer, now)` 纯函数，便于测试。
- `cmd/worker/main.go` 启动轻量 goroutine 定时器，每日 02:00 调用。

## 6. 测试策略

- 单测：
  - 存在训练/活动用户时入队并带正确 trigger 参数。
  - 空用户集合不入队。
  - 02:00 后启动时未执行则补跑；已执行则跳过。

## 7. 风险与回滚

- 风险：夜间执行与其他入队逻辑叠加导致队列堆积。
- 回滚：移除定时触发与相关函数即可。
