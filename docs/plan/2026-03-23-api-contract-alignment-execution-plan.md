# StrideWise 前后端 API 契约对齐执行计划（详细版）

更新时间：2026-03-23  
适用范围：`docs/frontend-backend-api-requirements.md`、`docs/2026-03-23-backend-internal-api-inventory.md`、`docs/api-integration-gap-analysis.md`

## 1. 背景与目标

当前主要问题不是“接口不存在”，而是“接口契约漂移”：
- 字段结构漂移：嵌套 vs 拍平、字段命名不一致。
- 枚举语义漂移：前端语义值与后端校验值不一致。
- 格式约束漂移：时间/配速/时长输入格式不一致。
- 范围漂移：后端已实现接口未进入前端需求清单。

本计划目标：
1. 建立单一契约事实源（OpenAPI）。
2. 将联调阻断项在最短路径内清零。
3. 建立可持续防漂移机制（契约测试 + CI 差异检查）。

## 2. 批判性结论（风险与隐含问题）

已识别的潜在风险：
1. 单靠人工维护三份文档会持续失真，后续迭代仍会复发。
2. 若前端继续自行映射复杂业务语义，会把后端领域规则分散到 UI 层，后期维护成本上升。
3. 当前“接口存在但需求文档未纳入”的模块会造成误判，影响排期和验收边界。

超出当前框架的建议（新增）：
1. 在 CI 引入“契约差异阻断”（requirements vs OpenAPI），把问题前置到合并前。
2. 建立“联调模式版本号”（例如 API-ALIGN v1），每次变更必须更新版本与变更单。

## 3. 执行范围与不做项

执行范围：
1. 推荐、训练、基线、用户档案接口契约对齐（MVP 首要）。
2. 同步/天气/趋势/训练反馈/系统接口纳入范围标记（已接入或暂不接入）。
3. 文档、代码、测试、CI 同步落地。

暂不做项：
1. 新增复杂业务接口（除为联调对齐必须的轻量接口）。
2. 权限体系升级（沿用内部 token 模式）。

## 4. 分阶段计划

## 4.1 阶段 A：契约冻结与版本化（P0，1 天）

任务：
1. 以 OpenAPI 文档定义单一事实源，设定版本 `v1.0.0-api-align`。
2. 将前端需求文档改为“引用 OpenAPI + 页面映射说明”，不再重复定义字段真值。
3. `gap-analysis` 只记录“偏差项与状态”，不再重复完整接口定义。

产出：
1. OpenAPI 契约版本段与变更记录。
2. 前端需求文档改造版。
3. gap-analysis 状态版（含阻断/非阻断）。

验收标准：
1. 三份文档字段真值一致。
2. 任意接口字段定义仅在 OpenAPI 出现一处主定义。

## 4.2 阶段 B：阻断项清零（P0，2-3 天）

任务：
1. 推荐接口对齐：
   - `latest` 输出支持前端可直接消费结构（或明确 BFF 映射）。
   - `rec_id` 与前端 `id` 命名对齐策略固定。
2. 推荐反馈对齐：
   - 固化 `useful/neutral/not_useful -> yes/neutral/no` 映射。
3. 训练日志对齐：
   - 训练类型映射表（英文 UI 枚举 -> 后端标准值）。
   - `pace/duration/start_time` 统一输入格式与校验错误码。
4. 用户档案对齐：
   - 增加目标字段局部更新能力，避免前端每次提交全量必填。
5. 基线接口对齐：
   - 输出前端可用结构（配速区间、建议周跑量、恢复级别），避免前端做领域推导。

产出：
1. 后端接口适配代码（必要处）。
2. 前端映射代码（必要处）。
3. 对齐后的请求/响应示例。

验收标准：
1. 阻断项列表为 0。
2. 主页推荐、训练新增、档案更新、基线展示四条主路径联调通过。

## 4.3 阶段 C：范围对齐与联调清单（P1，1 天）

任务：
1. 对后端已有接口做状态标记：`已接入/预留/暂不接入`。
2. 将同步、天气、趋势、训练反馈、系统接口补入“联调范围矩阵”。
3. 明确每个接口的前端消费页面与责任人。

产出：
1. 接口范围矩阵表（按模块）。
2. 联调优先级列表（MVP/Next/Backlog）。

验收标准：
1. 不再存在“后端已实现但文档无归属”的接口。
2. 前后端对同一接口状态认知一致。

## 4.4 阶段 D：自动化防漂移（P1，1-2 天）

任务：
1. 建立契约测试：
   - 后端：请求校验 + 响应 schema 测试。
   - 前端：类型生成与反序列化测试。
2. 建立 CI 差异检查：
   - requirements 文档中的接口集合必须为 OpenAPI 子集。
   - 枚举值偏差自动告警并阻断。

产出：
1. CI 脚本（契约对比 + 枚举校验）。
2. 契约测试报告模板。

验收标准：
1. 提交中引入契约漂移时 CI 必定失败。
2. 主分支保持可回放的契约一致性记录。

## 5. 任务拆解（可执行清单）

1. 文档治理任务
   - 更新 OpenAPI 为事实源并打版本。
   - 重写前端需求文档为“场景 + 映射”视角。
   - 更新 gap-analysis 为状态跟踪。
2. 后端任务
   - 推荐返回 DTO 适配。
   - 档案局部更新接口（或等效实现）。
   - 基线聚合响应 DTO。
   - 错误码标准化（`bad_request/conflict/...`）。
3. 前端任务
   - 推荐与日志映射层统一。
   - 格式化输入工具（pace/duration/time）。
   - 反馈枚举映射层。
4. 测试任务
   - 四条主路径集成测试。
   - 契约测试与快照。
5. CI 任务
   - 差异检测脚本接入流水线。
   - 失败信息可读化（显示具体字段差异）。

## 6. 优先级与里程碑

里程碑 M1（第 1 天结束）：
1. 文档契约冻结。
2. 阻断项与非阻断项边界明确。

里程碑 M2（第 3-4 天结束）：
1. 阻断项清零。
2. 四条主路径联调通过。

里程碑 M3（第 5-6 天结束）：
1. 范围矩阵完成。
2. 契约防漂移 CI 生效。

## 7. 质量门禁（Definition of Done）

1. 文档层：
   - 三份文档一致且可追溯到 OpenAPI 版本。
2. 代码层：
   - 主路径联调无阻断错误。
   - 关键映射逻辑有单测覆盖。
3. 流程层：
   - CI 可拦截契约漂移。
   - 每次接口变更必须更新版本与差异说明。

## 8. 风险与回滚策略

风险：
1. 一次性改动过大导致联调窗口拉长。
2. 前端页面模型变更影响现有展示逻辑。
3. 基线聚合字段定义反复变化。

控制策略：
1. 采用“新增字段向后兼容 + 分步切换”而非一次性破坏式修改。
2. 为关键接口保留旧字段过渡期（标注 deprecate）。
3. 对外暴露统一映射函数，禁止页面散落转换逻辑。

回滚策略：
1. 每个阶段单独提交，可按阶段回滚。
2. 契约版本支持回退到上一个稳定 tag。

## 9. 推荐实施顺序（落地建议）

1. 先做阶段 A（冻结契约），否则后续改动会反复返工。
2. 再做阶段 B（清阻断），确保 MVP 页面先跑通。
3. 然后做阶段 C（范围矩阵），避免“隐性欠账”。
4. 最后做阶段 D（CI 防漂移），把治理固化到流程。

## 10. 验收清单（供你后续调整）

1. 是否认可 OpenAPI 作为唯一字段真值来源。
2. 是否接受后端提供前端友好 DTO（减少前端业务映射）。
3. 是否接受新增“档案局部更新接口”以避免全量提交。
4. 是否将同步/天气/趋势/训练反馈标记为 Next 阶段接入。
5. 是否在当前迭代就启用 CI 契约阻断。
